# ADR-006: AI 상품 설명 — "설정"과 "상태" 필드 분리

- 상태: Proposed
- 날짜: 2026-04-23
- 결정자: @이재범 (제안), @팀 (논의 대기)

---

## 맥락

현재 ERD에 `product.is_ai_description` 필드가 하나 있다. 그런데 이 필드가 **두 가지 의미를 동시에** 가지고 있어 혼선을 일으킨다.

- **"설정" (intent)**: "사장님이 이번에 AI 설명 기능을 쓸 것인가"
- **"상태" (state)**: "저장된 이 설명이 실제로 AI가 쓴 것인가"

### 예시로 드러난 문제

```
1) 사장님이 상품 등록 시 AI로 설명 생성 → is_ai_description = true
2) 나중에 설명을 직접 수정
3) 그런데 is_ai_description은 여전히 true로 남음
   → "지금 이 설명은 AI가 쓴 건가? 사장님이 고친 건가?" 구분 불가
```

필드 하나가 **요청 시점의 의도**와 **저장된 결과의 사실**을 동시에 표현하려다 보니, 수정이 들어오는 순간 정합성이 깨진다.

---

## 결정

**"설정"과 "상태"를 다른 필드로 분리**한다. 추가로, 사장님의 "AI 기능을 쓸지 말지" 전역 설정은 User 테이블로 옮긴다.

### 필드별 역할

| 위치 | 필드명 | 의미 | 시점 |
|---|---|---|---|
| `User` 테이블 | `use_ai_description` (TBD 네이밍) | 사장님이 AI 기능을 쓰기로 했는지 (**전역 설정**) | 마이페이지 설정 |
| `Product` 테이블 | `is_ai_description` | 현재 저장된 이 설명이 AI 생성물인지 (**저장된 결과**) | 등록/수정 시점 확정 |
| `ProductRequest` DTO | `autoGeneratedDescription` | 이번 요청에서 AI 호출을 쓸지 (**요청 의도**) | 요청마다 선택 |

> 같은 "AI 설명"이라도 **어느 레이어에 있느냐**에 따라 이름·의미가 달라진다.

### 요청 DTO 의미

```java
// ProductCreateRequest / ProductUpdateRequest
autoGeneratedDescription: true    // "이번엔 AI가 써줘"
autoGeneratedDescription: false   // "내가 직접 쓸게"
autoGeneratedDescription: null    // "기본값으로 해" (User.use_ai_description 참조)
```

### 서버 처리 로직

```
if (request.autoGeneratedDescription == true):
    description = geminiClient.generate(prompt)
    product.is_ai_description = true
elif (request.autoGeneratedDescription == false):
    description = request.description        // 사장님이 직접 쓴 값
    product.is_ai_description = false
elif (request.autoGeneratedDescription == null):
    // User.use_ai_description 기본값으로 분기 (TBD)
```

## 케이스별 동작

| # | 상황 | RequestDto | 서버 동작 | 저장 결과 |
|---|---|---|---|---|
| 1 | AI로 설명 생성 | `autoGenerated=true`, `description=null` | AI 호출 → description 채움 | `is_ai_description=true` |
| 2 | 직접 작성 | `autoGenerated=false`, `description="우리집 특제 치킨"` | AI 호출 X | `is_ai_description=false` |
| 3 | AI로 만들었던 것을 나중에 수정 | `autoGenerated=false`, `description="수정된 설명"` | AI 호출 X | `is_ai_description=false` (사람이 고쳤으므로) |
| 4 | 설명 없이 등록 | `autoGenerated=false`, `description=null` | AI 호출 X | `description=null`, `is_ai_description=false` |

---

## 대안

| 대안 | 기각 이유 |
|---|---|
| **A. 현재 그대로 `is_ai_description` 하나로 혼용** | 수정 시점에 의도와 결과가 어긋나면 정합성 깨짐 (위 "예시로 드러난 문제") |
| **B. 필드 분리: User(전역 설정) / Product(저장 결과) / RequestDto(요청 의도)** ✅ 채택 | 각 필드가 한 가지 의미만 가지므로 충돌 없음, 히스토리 추적 가능 |
| **C. `ai_generated_at` 타임스탬프만 두기** (AI 생성 시점만 기록) | 수정 여부 판단이 여전히 모호, 현재 상태를 질의할 때 매번 비교 필요 |

---

## 결과

### 👍 좋은 점

- **의미 충돌 제거**: "요청 의도"와 "저장된 결과"를 같은 필드로 섞지 않음
- **수정 이력 자연스럽게 반영**: 사람이 고치면 `is_ai_description=false`로 자동 갱신
- **전역 설정 분리**: AI 기능을 쓰기로 했는지를 사장님 단위로 저장 가능 (user 테이블)
- **기본값 처리 명확**: 요청에서 `null`이면 사용자 전역 설정을 따라감

### 👎 감수한 트레이드오프

- **필드 3개로 늘어남**: User, Product, RequestDto 각각 다른 필드가 필요 → 팀원 학습 비용 소폭 증가
- **네이밍 혼동 가능**: "AI 설명" 관련 필드가 3개라 이름 규칙을 팀에서 맞춰야 함
- **AI 생성 히스토리는 별도로 관리 필요**: 이 설계는 "현재 상태"만 기록. "예전에 AI로 만들었다가 고친 이력"을 보려면 `p_llm_calls` 로그를 조회해야 함

## 적용 범위

### 데이터 모델 변경

- [ ] `p_user` 테이블: `use_ai_description` (`BOOLEAN`, default `false`) 컬럼 추가
- [ ] `p_product` 테이블: 기존 `is_ai_description` 유지 (의미만 "저장 결과"로 고정)
- [ ] [데이터 명세](../design/data.md) / [도메인 명세](../design/domain.md) 업데이트 필요

### 요청 DTO

- [ ] `ProductCreateRequest`, `ProductUpdateRequest`에 `autoGeneratedDescription: Boolean?` 필드 추가 (`null` 허용)

### 서비스 로직

- [ ] ProductService에서 `autoGeneratedDescription` 분기 처리
- [ ] `null`이면 User 전역 설정(`use_ai_description`) 참조 후 분기

### 네이밍 확정 (TBD — 팀 확정 필요)

- `User.use_ai_description` vs `ai_description_enabled` vs `ai_description_opt_in`
- `autoGeneratedDescription` vs `useAiDescription` vs `generateWithAi`
